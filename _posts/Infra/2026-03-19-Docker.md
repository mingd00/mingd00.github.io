### What is Docker?

> A technology that **packages applications and infrastructure** together to ensure **consistent execution in any environment**.

- **application**: code with logic
- **infrastructure**: the environment for execution of the application

The application always depends on the infrastructure.

Errors can occur even when running the same code if the server or runtime version is different, and a lot of resources are wasted to align these infrastructure environments.

Docker solves this problem.

Docker packages these two into a single container and **eliminates the difference between the development and execution envirnoments**.

<br>

### Docker Features

- **immutable infrastructure**: the state of a server at a specific point in time can be saved and replicated
- **portability**: consistent results are produced across different environments
- **idempotency**: the result does not change even if the same operation is repeated multiple times
- **easy management**
- **fast deployment speed**

<br>

### The deployment process of a Docker container

> "Create a Dockerfile -> Build an Image -> Run a Container"

Docker manages applications and infrastructure together in a single container.<br>
An image is a template for creating a container, and the build process of the image is defined in a Dockerfile.

<br>

### Dockerfile

> The step to build an image

```python
FROM python:3.10 # base image (from docker hub)

COPY . /app # the path to copy into the Docker container

RUN pip install -r requirements.txt # the command to execute when building the image

CMD ["python", "app.py"] # the command to execute when the container starts
```

<br>

### Image

> A template for creating a Docker container

- There are 2 ways to create an image

#### 1. build directly using a Dockerfile <br>
    ```python
    docker image build -t [image-name]:[tag-name] [Dockerfile-path] 

    # custom docker file
    docker image build -f [Dockerfile-name] -t [image-name]:[tag-name] [Dockerfile-path]
    ```

#### 2. pull a public image from docker hub <br>
    ```python
    docker image pull [repository-name]:[tag-name]
    ```

- check images
```python
docker images
```

<br>

### Container

> A virtual box that includes applications and infrastructure.

After the image build is complete, the container should be run to operate the application.<br>

Port forwarding is important because a Docker container runs in an isolated environment and has its own **internal container port**. Therefore, it is necessary to connect it to a **host machine port** to allow external access.<br>

It is also important to specify the image name and tag together for easier version management.<br>

- run

    ```python
    docker container run -d -p 9000:8000 [image-name:tag-name]

    # p 9000:8000 <- port forwarding(host-port : container-port)
    ```

- stop

    ```python
    docker container stop [container-name]
    ```

- remove

    ```python
    docker container rm [container-name]
    ```

- remove all stopped containers

    ```python
    docker contanier prune
    ```

- check logs

    ```python
    docker container logs [container-name]
    ```

- execute commands inside the container

    ```python
    docker container exec -it [container-name] sh
    ```

- copy

    ```python
    # container ->  host
    docker contanier cp [container-name]:[origin-path] [target-path]

    or

    # host -> container
    docker container cp [origin-path] [container-name]:[target-path]
    ```

- check containers

    ```python
    docker ps

    # including stopped containers
    docker ps -a
    ```

### docker-compose

> Manage multiple container executions at once

Building images and running containers with a single commands is possible using Docker Compose. <br>
Various settings can be defined in a single file. It allows you to set up the environment with one command and manage multiple images and containers in one file.<br>

In real projects, using the `docker run` command is rare. Docker Compose is usually used to set up the environment.

```python
services:

  embedding-server: # define a service
    build: . # build an image using a Dockerfile(specify the Dockerfile path)
    image: embedding-server:latest # image-name:tag-name
    container_name: embedding-server # container-name
    ports:
      - "8000:8000" # port forwarding -> host-port:container-port
    restart: unless-stopped # automatically restart unless stopped manually
    volumes: # volume mount -> host path:container path connect
      - /data/embedding:/app/data
    depends_on: # set dependencies
	    - postgresql # run after postgresql is running
      
  postgresql:
    image: postgres # use official postgres image 
    restart: always # restart under any condition
    container_name: postgres
    ports:
      - "5432:5432"
    environment: 
      POSTGRES_DB: postgres
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres 
      PGDATA: /var/lib/postgresql/data/pgdata # internal path for storing actual data
    volumes:
      - /data/postgres_260226:/var/lib/postgresql/data
```

- run: `docker-compose up -d` (force build option: `--build` )
- stop: `docker-compose down`
